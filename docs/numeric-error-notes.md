# 計算誤差解説ノート

## なぜ 0.28 - 0.27 が 0.01 にならないのか

コンピュータは小数を **2進数（IEEE 754 浮動小数点数）** で近似して記憶します。

たとえば `0.28` を2進数で表すと、無限に続く循環小数になります。
コンピュータはこれを有限のビット数（64ビット）に切り捨てて保存するため、
**微小な誤差**が生まれます。

```
0.28 - 0.27 の実際の結果:
  0.010000000000000001  ← 0.01 ではない
```

## 誤差が起きやすい演算

| 演算 | 例 | 期待値 | 実際の値 |
|------|-----|--------|---------|
| 減算 | `0.28 - 0.27` | `0.01` | `0.010000000000000001` |
| 加算 | `0.1 + 0.2` | `0.3` | `0.30000000000000004` |
| 比較 | `0.1 + 0.2 == 0.3` | `True` | `False` |

## 回避策

### 1. 整数スケール法（推奨）

小数点以下の桁数に合わせて整数倍し、最後に割る。

```python
# 誤差あり
result = 0.28 - 0.27        # 0.010000000000000001

# 整数スケール法（誤差なし）
result = (28 - 27) / 100    # 0.01（正確）
```

金額計算では **円（整数）** で管理し、表示時だけ小数にするのが定石です。

### 2. 誤差の許容比較

厳密な一致ではなく、「ほぼ等しい」で比較する。

```python
diff = abs(result - 0.01)
if diff < 0.0001:
    print("ほぼ 0.01 です")
```

## 授業での説明のポイント

- 誤差は「バグ」ではなく「仕様」（有限ビット表現の限界）
- 10進数の `0.1` は2進数で表現できない（`1/3` が10進数で表現できないのと同じ）
- 実用上は「整数で管理する」「誤差許容比較を使う」で対処できる
- プロの開発者も常に意識する重要な概念
